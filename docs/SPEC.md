# SPEC.md — архитектура и модель данных

Источник: дизайн Plan-агента, утвержден 2026-07-18. Схема реализуется в `packages/db` (Drizzle), инкрементально по фазам. Статус реализации помечен: [Ф0 ✅] — уже в коде, остальное — целевые блоки.

## 0. Сквозные принципы

- **PK**: UUID (v7 генерируется приложением при вставках) — безопасное слияние инстансов в облако, отсутствие конфликтов при импорте.
- **tenant_id NOT NULL** на всех таблицах, уникальные индексы составные с tenant_id. Деплой single-tenant; RLS не включаем (готовность в схеме, не в рантайме).
- **Версионность** справочников (тарифы, нормативы, ставки) — `valid_from date / valid_to date NULL`; в PostgreSQL v2 — EXCLUDE-констрейнты (btree_gist) на непересечение; в PGlite v1 непересечение контролируется приложением при сохранении.
- **Деньги** `numeric(14,2)`, **тарифы** `numeric(12,4)`, **объемы/нормативы** `numeric(14,6)`. Округление — только в итогах строк, правила фиксируются в следе расчета.
- **Историчность фактов** (жильцы, владельцы ЛС, статусы ПУ) — интервальные записи `date_from/date_to`, не перезапись.
- **Append-only** для начислений закрытых периодов, платежей, аудита; прошлое меняется только корректирующими документами в текущем открытом периоде.

### C. Абоненты и лицевые счета
Решение: **помещение и ЛС — раздельные сущности.** Смена собственника = закрытие старого ЛС (долг остается на нем) + открытие нового с датой; пропорция месяца — из date_open/date_close. Разделение ЛС = закрытие исходного + N новых с долями.

| Таблица | Ключевое | Комментарий |
|---|---|---|
| `customer` | физ/юр: ФИО+документы+СНИЛС / название+ИНН+КПП+ОГРН+банк | CHECK по kind; ПДн — маскирование в логах |
| `account` [Ф0 ✅ ядро] | номер ЛС (unique per tenant), kind (residential/legal), premise_id/contract_id, date_open/close, close_reason, predecessor_account_id, ownership_share, gis_els, gis_zhku_id | цепочка predecessor при смене собственника/разделении; ЕЛС/ЖКУ-ид присваивает ГИС ЖКХ (обратный импорт) |
| `account_customer` | роль (owner/tenant/payer) с датами | текущий плательщик — для квитанции |
| `account_resident` | зарегистрированные: ФИО, вид (пост./врем.), date_from/to | база норматива ГВС; число на каждый день месяца — из интервалов |
| `absence_claim` | временное отсутствие (раздел VIII ПП 354): даты, документы, статус, кто утвердил | порождает авто-перерасчет ГВС; отопление не пересчитывается |
| `benefit_category` | код ЕГИССО, % скидки, услуги, mode: discount_in_bill / registry_only | два режима: скидка в квитанции vs реестр для соцзащиты |
| `account_benefit` | льгота на ЛС/жильца с датами | пропорция дней |

## 2. Ключевые механики

**Закрытие/переоткрытие периода.** Статусная машина, переходы — отдельные операции с проверкой роли и аудитом. Закрытие в одной транзакции: контрольная сверка (суммы строк = шапкам, все ЛС посчитаны) → снапшот account_balance → status=closed. Переоткрытие — аварийная операция администратора, только последнего закрытого периода, с причиной в аудит.

**Идемпотентный пакетный расчет 10к ЛС ≤ 60 с (v2; v1 — те же шаги в Web Worker).** (1) set-based выгрузка снапшота 6–8 запросами (ЛС+дома, интервалы жильцов, показания и статусы ПУ, версии тарифов/нормативов, пересеченные с границами месяца); (2) чистое ядро без I/O, параллель по чанкам ~500 ЛС (1–5 мс/ЛС); (3) COPY-вставка одной транзакцией. Повторный запуск помечает прежние accrual scope is_current=false. Ошибка одного ЛС → в stats, тираж продолжается.

**Фоновая генерация PDF.** Чанки по 200–500 квитанций, прогресс в background_job; выход: индивидуальные PDF + сшитые пачки по участкам. Быстрый движок (в v2 — Typst/ReportLab-класс производительности; НЕ WeasyPrint — 0.5–1 с/док).

**Конкурентный ввод показаний.** Partial UNIQUE (meter, period, accepted) в БД; второй оператор получает конфликт с выбором «оставить/перезаписать» (перезапись = version+1, старая строка rejected, всё в аудит). Валидация «не меньше предыдущего» в транзакции с учетом перехода через 0 и замены ПУ.

**Аудит и след расчета.** Аудит — append-only (v2: партиции + REVOKE). calc_trace в каждой строке: `{formula, inputs: {S_i, S_об, V_д, дни, версии тарифов…}, steps, rounding}` + engine_version в шапке → кнопка «объяснить расчет» рендерит трейс человеку; ответ в ГЖИ без повторного расчета.

**Импорт реестров оплат.** Файл (sha256-дедуп) → парсер профиля формата → payment_row с raw → матчинг по номеру ЛС (fallback ФИО+адрес) → matched/quarantine/duplicate_suspect. Частичное принятие: matched транзакционно → payment+allocation (ФИФО); карантин — UI разбора; итог реестра сходится: rows_total = matched + quarantined + rejected.

## 4. Риски архитектуры (топ-5) и гашение

1. **Краевые случаи ПП 354** (смена тарифа + собственника + поверка в одном месяце): чистое ядро + golden-тесты из реальных квитанций пилота + единый механизм сегментации месяца + обязательный trace + параллельный прогон.
2. **Качество данных пилота** (Excel): импорт первой фазой с валидационным отчетом и карантином; входящее сальдо — отдельный выверенный документ с подписью МУП.
3. **Регуляторная изменчивость**: версионные reg_parameter/key_rate, template_version у экспортов, ГИС-адаптер изолирован; формы квитанций — данные+шаблон.
4. **Целостность финансов при конкурентности**: инварианты в БД (partial unique, статусы, снапшот-сальдо), автосверка контрольных сумм при закрытии.
5. **Один разработчик + рост объемов**: perf-тест 10к ЛС в CI с бюджетом, партиционирование accrual_line/audit_log заложено в v2-миграции, компактный числовой trace, ADR-записи.
