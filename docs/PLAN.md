# «Теплобиллинг» — продукт начислений ЖКХ (отопление + ГВС) для МУП/РСО

## 1. Контекст

Реальный МУП «Тепловые сети» (~10 000 ЛС) прислал ТЗ на модуль начислений (отопление + ГВС по ПП №354). Пользователь строит это как **свой тиражируемый продукт**, МУП — пилот и первый клиент. ТЗ содержит серьезные пробелы (нет платежей, пеней, ГИС ЖКХ, юрлиц-методики, упрощенные формулы) — все выявлены, verified веб-ресёрчем (8 агентов) и закрыты решениями ниже.

**Форма первой версии (решение пользователя 2026-07-18): v1 полностью работает в браузере (как СвояКнига), код на его GitHub, живая ссылка через GitHub Pages — тестирует сам и даёт МУПу. Позже — VPS.**

- ТЗ: `C:\Users\user\Downloads\ТЕХНИЧЕСКОЕ ЗАДАНИЕ.docx` (= вставленному тексту). Excel п.11 отсутствует — запросить у МУП, до тех пор демо-данные.
- Проект: `D:\teplobilling\` (правило: проекты на D:). GitHub: аккаунт A1exxx, repo `teplobilling`, Pages `a1exxx.github.io/teplobilling/`. Repo public (как остальные проекты юзера; формулы — из закона, секретов нет; при коммерциализации можно увести в private — пункт на потом).
- Полные материалы агентов (модель данных ~45 таблиц, каталог расчетных случаев, ресёрч): task-outputs сессии; при старте реализации переносятся в `docs/SPEC.md`, `docs/CALC-SPEC.md`, `docs/RESEARCH.md` репозитория. Существенное продублировано в этом плане.

## 3. Стек v1 → v2 (без переписывания)

**TypeScript везде** — вынужденно и выгодно: v1 живет в браузере (Python там не живет), v2 на VPS переиспользует то же ядро на Node. Отступление от ТЗ (Python/C#) — осознанное, ТЗ писалось под десктоп.

| Слой | v1 (браузер, GitHub Pages) | v2 (VPS) |
|---|---|---|
| БД | **PGlite** (настоящий PostgreSQL в WASM), персист в IndexedDB + экспорт/импорт файла БД (бэкап руками) | PostgreSQL 16 — та же схема |
| ORM/миграции | **Drizzle ORM + drizzle-kit** (официально поддерживает и PGlite, и node-postgres — одна схема на оба мира) | тот же Drizzle |
| Ядро расчета | `packages/billing-core` — чистый TS, decimal.js (float запрещен), без DOM/IO | тот же пакет на Node |
| UI | React 19 + Vite + **Ant Design v6** (плотные учетные интерфейсы) + **refine.dev v5** (headless admin: CRUD, RBAC, audit — MIT) | тот же фронт, data-provider меняется с PGlite на REST |
| Грид показаний | **Glide Data Grid (MIT)** — вставка диапазона из Excel работает (AG Grid Community — нет, verified; Handsontable платный) | тот же |
| Тяжелые расчеты | Web Worker-пул (не блокировать UI; 10к ЛС укладывается) | Node worker_threads / очередь в PG (SKIP LOCKED) |
| PDF-квитанции | print-CSS (печать из браузера) + @react-pdf/renderer или pdfmake со вшитыми кириллич. шрифтами; QR: библиотека `qrcode` + строка ST00012 по ГОСТ Р 56042-2014 (портируем kiraind/gost-r-56042-2014-js, MIT; **Sum — в копейках!**) | серверная пакетная генерация 10к PDF фоном |
| Excel | SheetJS CE / exceljs в браузере (импорт реестров, ведомости, шаблоны ГИС ЖКХ, выгрузка 1С) | то же на сервере |
| Деплой | GitHub Actions → gh-pages | docker-compose (Linux VPS): app, postgres, nginx, worker |
| Монорепо | pnpm workspaces: `packages/billing-core`, `packages/db` (drizzle-схема), `apps/web`; позже `apps/server` (Fastify) | |

Ограничения v1, проговорить в README и МУПу: однопользовательский режим (данные в браузере конкретного компьютера; экспорт/импорт файла БД для переноса), роли условны, массовая печать 10к PDF и ГИС-SOAP — с v2. Для пилотного теста и продажи — достаточно.

## 4. Готовые решения (итог ресёрча, 8 агентов, verified)

**Целиком готового open-source биллинга ЖКХ НЕТ** — ни на GitHub, ни в PyPI/npm; открытого шаблона квитанции по форме Минстроя тоже нет. Берем кусками:
- **QR ГОСТ Р 56042-2014**: kiraind/gost-r-56042-2014-js (MIT, TS, ~30 полей) — использовать/портировать. Вердикт verified: поле Sum в копейках.
- **ГИС ЖКХ**: gosuslugi-api (python, read-only справочники) и C#-паттерны XAdES-подписи (RIP-webmaster/CommunalServices, gizmo75rus/HCS — GPL, только как референс) — пригодятся в v2 для SOAP-модуля; сам SOAP-контур пишется свой.
- **1С-обмен**: kilylabs/client-bank-exchange-php (MIT, формат 1CClientBankExchange для реестров банка) — портировать парсер формата на TS.
- Учебные репо с формулами 354 — только как перекрестный референс тестов.

**Рынок (buy-vs-build, честно)**: готовые решения для одного МУП в 4–10 раз дешевле заказной разработки — Стек-Диво (до 10к ЛС) от 2 900 ₽/мес, РосКвартал от 0,8 ₽/ЛС/мес, 1С:Расчет квартплаты от 38 900 ₽, 1С:Управление теплосетью 2 — 364 500 ₽ (verified 18.07.2026). Следствие для нашего продукта: **конкурируем не ценой, а UX-ом, скоростью внедрения, теплоспецификой (двухкомпонентный ГВС, юрлица-теплосеть) и прозрачностью расчета (trace каждой цифры)**. Чеклист must-have зрелого биллинга (25 пунктов: от начислений и пеней до судебной работы, ЛК абонента, СБП, соцзащиты) — сохранить в docs/ROADMAP.md как северную звезду; v1 закрывает ~60% пунктов.

## 5. Юридическая база ядра (verified 2026-07-18; все — в код как версионируемые правила, не хардкод)

Исправления к ТЗ и уточнения агентов (двойная проверка research+verify):
1. Отопление, дом с ОДПУ: три ветки по ИПУ — ни одного (ф.3), часть (ф.3(1), появилась после ПКС №30-П от 10.07.2018 + ПП 1708), все (ф.3(3)).
2. Без ОДПУ: ф.2 — оплата в отопительный период, ф.2(1) — равномерно с коэффициентом K (различие — способ оплаты, НЕ тип здания); способ выбирает субъект РФ; K = месяцы отопительного периода / 12, только из НПА.
3. **Годовая корректировка отопления — ф.3(4)** (ф.3(2) утратила силу с 01.01.2019), обязательна в I квартале следующего года, может быть отрицательной.
4. Индивидуальные источники тепла (после ПКС №46-П): личное потребление 0, платится доля на общее имущество.
5. **Кпов 1,5**: только ГВС (и только компонент холодной воды в двухкомпонентном тарифе), при техвозможности установки ИПУ; к отоплению НЕ применяется (отменен ПП №232 от 27.02.2017); ХВС с 06.12.2025 — 3,0 (ПП №1871), нас не касается — не путать.
6. Непередача показаний: п.59 — среднемесячное за ≥6 мес, максимум 3 периода подряд; п.60 — далее норматив (+Кпов где положен). Истечение поверки/поломка — тот же режим со дня события (месяц делится по дате).
7. Двухкомпонентный ГВС: ф.23 (индивид.) — P = V×T_хв + (V×q_кр)×T_тэ; q_кр — региональный норматив подогрева (~0,05–0,07 Гкал/м³), НЕ фактический удельный расход (позиция ВС РФ — ловушка №1 двухкомпонентников); открытая система — компонент «теплоноситель»; ОДН при двухкомпонентном — ф.24; ф.25–27 — коммуналки.
8. Ноль зарегистрированных без ИПУ — по числу собственников (п.56(2)).
9. Временное отсутствие (раздел VIII): >5 полных дней, только услуги без ИПУ при акте о невозможности установки, НЕ отопление никогда; заявление до отъезда или ≤30 дней после; исполнить за 5 рабочих дней.
10. Пени физлиц (ст.155 ч.14 ЖК): 0 (1–30 дн) / 1/300 (31–90) / 1/130 (91+); ставка = min(ключевая на день оплаты; 9,5%) — льгота ПП №329 от 18.03.2025 действует до 01.01.2027, НЕ хардкодить; мораторные окна 2020/2022 — данными. Юрлица: ФЗ-190 ст.15 ч.9.1–9.3 — прочие потребители 1/130 с 1-го дня; УК/ТСЖ — 1/300→1/170→1/130.
11. **177-ФЗ от 24.06.2025 (действует с 01.03.2026): единый срок оплаты — до 15-го числа** (было 10-е) → отсчет пеней от нового срока; версионируемый параметр.
12. ГИС ЖКХ: обязанность РСО — ст.6 209-ФЗ; состав/сроки — **приказ Минстроя №79/пр от 07.02.2024 (ред. 20.11.2025)** (заменил 74/114/пр); размещать: ЛС, ПД/начисления, показания, задолженность+исполнительные производства (новое по 177-ФЗ); штраф юрлицу до 200 000 ₽ (ст.13.19.2 КоАП).
13. Квитанция: обязательные реквизиты — п.69 ПП 354; примерная форма — приказ №43/пр (действует, рекомендательная); QR по ГОСТ Р 56042-2014 обязателен (для РСО отказ невозможен); ЕЛС (10 знаков) + идентификатор ПД ГИС ЖКХ (18 знаков).
14. ПП 354 в текущей редакции действует до 31.12.2027 (регуляторный эксперимент) → формулы будут переиздаваться; лечится версионированием методик (ниже).

## 6. Расчетное ядро `billing-core` (дизайн принят)

Полный каталог случаев — в выводе Plan-агента (перенести в docs/CALC-SPEC.md): отопление H-01…H-13, ГВС W-01…W-11 + D-1/D-2, общие G-01…G-06, юрлица L-01…L-06, пени. Принципы:
- **Чистый пакет**: вход — frozen-контекст со ВСЕМИ данными (интервальные тарифы/нормативы/ставки/жильцы/состояния ПУ), выход — строки с полным trace (формула, входы, шаги, округления). Времени «сейчас» в ядре нет (запрещено линтером).
- **Селектор формулы** — один декларативный список правил (предикаты → MethodId) с машиночитаемым обоснованием в trace; тестируется полной таблицей решений.
- **Версионирование методики**: редакция формулы = отдельный класс с датами действия; расчет за январь-2025 из 2027 года — бит-в-бит тот же. Данные (Кпов, cap 9,5%, срок оплаты 10-е→15-е, мораторные окна) — версионируемые интервалы.
- **DateSplitter**: месяц → интервалы постоянства всех параметров (смена тарифа/жильцов/статуса ПУ/собственника) → подстроки; фактические дни месяца, полуинтервалы [from,to).
- **Округление**: Decimal; промежуточные не округлять; итог строки — копейки ROUND_HALF_UP один раз; распределение ОДПУ по дому — метод наибольших остатков (Σ долей == V_д точно, property-тест).
- **Пени** — интервальный event-движок по требованиям (месячным долгам): таймлайн событий (срок оплаты по закону периода, дни 31/91, частичные оплаты уменьшают базу со следующего дня, смены ставки/cap, мораторные окна, перерасчеты базы) → интервалы постоянства → Σ. Никаких «долг×дни×ставка».
- **Перерасчеты** = replay+diff: закрытые периоды immutable; пересчет истории по правилам того месяца → дельта документом текущего периода; идемпотентность (повторный replay → дельта 0) — property-тест.
- **Тесты**: ~120 golden-кейсов YAML (эталоны: калькулятор ГИС ЖКХ/ФАС ≥30 шт., разборы ГЖИ, двойной ручной расчет), property-based (fast-check), мутационные (Stryker) для methods/penalty, табличный тест селектора, 100% branch coverage ядра в CI-гейте.
- Юр-сверка всех пометок [проверить] по действующей редакции ПП 354 — отдельная задача перед кодированием каждой формулы, результат — в докстринге (пункт + дата сверки).

## 7. Модель данных (~45 таблиц, дизайн принят; полная спека → docs/SPEC.md)

Блоки: A адресный реестр с ГАР-кодами (нужны ГИС ЖКХ) → B жилфонд (дом: площади, МОП, тип системы ГВС, категория норматива, способ оплаты отопления; сезоны) → C абоненты (ЛС ≠ помещение; жильцы интервалами; льготы 2 режимов; временное отсутствие) → D приборы (ИПУ/ОДПУ/узлы юрлиц; поверки; показания с UNIQUE accepted и источником estimate_avg/estimate_norm) → E версионные тарифы/нормативы/ключевая ставка/рег-параметры (EXCLUDE-констрейнты непересечения) → F договоры юрлиц (точки поставки, границы балансовой принадлежности, объемы эт.1, нагрузки эт.2) → G расчетный контур (billing_period: open→calculating→calculated→approved→closed, триггер-запрет DML в закрытый; calc_run; accrual/accrual_line с calc_trace JSONB; recalc_doc) → H деньги (реестры с карантином строк, платежи, разнесение ФИФО ст.319.1 ГК, сальдо = immutable снапшот при закрытии) → I пени (транши + исключения/моратории) → J пользователи/аудит(append-only)/фоновые задачи/файлы/квитанции (qr_string сохраняется)/ГИС-выгрузки (версия шаблона).

Сквозное: PK UUIDv7, tenant_id везде, историчность интервалами, деньги numeric(14,2)/тарифы (12,4)/объемы (14,6).

- Перф: синтетика 10 000 ЛС в Web Worker — замер в CI (браузерный smoke через Playwright); жесткий гейт ≤60с — на серверной v2.
- E2E (Playwright): оператор вводит показания → расчет → квитанция → импорт оплат → закрытие; прогон на каждом релизе.
- Пилот (v2): 2–3 месяца параллельного расчета со старой системой, 100% расхождений классифицированы.

## 11. Риски

1. Ошибки в формулах → массово неверные квитанции: golden+property+мутационные тесты, trace, юр-сверка каждой формулы перед кодированием, параллельный прогон.
2. ПП 354 переиздадут (действует до 31.12.2027): версионируемые методики и параметры — новые классы правил, не правки.
3. Браузерная v1 ≠ многопользовательская реальность МУП: честно позиционировать как демо/пилотный контур; v2 — сервер. Экспорт файла БД = переносимость данных с первого дня.
4. Рынок конкурентен и дешев (Стек от 2 900 ₽/мес): позиционирование — UX, теплоспецифика, прозрачность расчета; решение о цене — после пилотного фидбека.
5. Грязные данные пилота (Excel): импорт с валидационным отчетом и карантином — уже в Ф1.
6. Продажи по 44-ФЗ без юрлица РФ: открытый бизнес-вопрос, эскалирован пользователю.
